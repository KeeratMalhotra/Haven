# Calendar key fix, smart fetch scoping, AI chat detach/attach, Spotify vertical drag

This change addresses four separate UX concerns: calendar TimeBlock keys now incorporate `event.start` so React re-mounts on drag (making the time text update), fetch scoping is narrowed per view (Day=1, Week=7, Month=30 days), the AI chat panel gains a detach/attach mode (slide-from-right or free-floating draggable), and the Spotify mini-player button becomes vertically draggable with localStorage persistence. A dashboard unauthenticated-state redirect is also added.

**Watch for:** The detached chat panel's `onClose` does not reset `detached` state, meaning re-opening always returns to detached mode once selected (confirmed). Navigation within a view (next/prev day/week) doesn't re-fetch events because `currentDate` is missing from the useEffect deps (confirmed). The Spotify button's drag-vs-click discrimination relies solely on Framer Motion's internal threshold with no explicit distance guard, so quick vertical drags can fire `handleExpand` (likely).

## High-level view

The TimeBlock key fix (`key={event.id}-${event.start}`) forces React to destroy and recreate the component when an event is dropped onto a new slot, which makes the displayed time text update. The approach depends on `setEvents` propagating the new ISO strings before reconciliation — the coupling between key format and drag-id format is fragile if either is changed independently.

Smart fetch scoping keys the `useEffect` dependency on `view`, so switching from Month to Day re-fetches with a narrower window. Navigating forward/backward within the same view doesn't trigger a re-fetch because `currentDate` is not in the dependency array.

The AI chat panel splits into two rendering paths: attached (slide-from-right with backdrop) and detached (free-floating draggable, centered on screen). The detached window has no drag constraints, so it can be moved entirely offscreen with no recovery mechanism.

The Spotify button switches from a static `<motion.button>` to a `<motion.div>` with `drag="y"` and computed constraints. Position is persisted to localStorage and restored on mount. The previous snap-threshold logic is removed.

1. **Navigation doesn't re-fetch events** — `currentDate` is not in the `useEffect` dependency array for fetching events, so navigating forward/backward within the same view serves stale data. Add `currentDate` (or a derived date-range key) to the deps.
2. **Detached state persists after close** — closing the chat panel via `onClose` sets `chatOpen=false` but leaves `detached=true`, so reopening always shows the floating window. Either reset `detached` on close or persist it intentionally with localStorage.
3. **Detached panel has no drag constraints** — the floating chat window uses `drag` without `dragConstraints`, so users can move it entirely offscreen with no way to recover it short of re-attaching (which requires the title bar to be visible). Add viewport constraints or a "reset position" escape hatch.
4. **Spotify button click fires on drag release** — the `onClick={handleExpand}` handler on the same element that has `drag="y"` will fire after short drags because there's no explicit activation distance. Use `onTap` instead or gate `onClick` on whether a drag occurred.
5. **Spotify button constraint uses stale window.innerHeight** — `maxDragUp` and `maxDragDown` are computed at render time from `window.innerHeight`, which won't update on viewport resize. The button could end up outside the visible area after an orientation flip or window resize.

## Calendar key change and time-text update mechanism

The key construction `${event.id || event.summary}-${event.start}` means that when `handleDragEnd` calls `setEvents(prev => prev.map(...))` with a new `start` ISO string, React sees a different key and unmounts the old TimeBlock, mounting a new one that reads the updated times.

The `draggedEventId` is constructed as `event-${event.id || event.summary}-${event.start}`. After a drop, the state update changes `event.start`, which means the `isDragging` check will never match the new event. This works because `handleDragEnd` sets `draggedEventId` to `null` synchronously before re-render. However, the coupling between key format and drag-id format is fragile — a future refactor that changes one without the other would break drag opacity feedback.

## Smart fetch scoping misses navigation

```tsx
useEffect(() => { ... }, [accessToken, view]);
```

The fetch fires when `view` changes but not when the user navigates (clicking "next week" or "previous day"). `currentDate` is not a dependency. Switching from Week to Day re-fetches correctly, but clicking "next" three times within Day view keeps showing whatever was fetched for the original date range. Since `fetchCalendarEvents(accessToken, days)` likely fetches relative to "now" on the backend, navigating away from today would show no events even if they exist in the target range.

## Detached chat panel — no constraints, sticky state

The detached panel renders with `drag` and `dragMomentum={false}` but no `dragConstraints`. Users can push the window entirely below the fold or past the left/right edges. If the title bar goes offscreen, the "Minimize2" re-attach button becomes unreachable.

The `onClose` handler in the layout is `() => setChatOpen(false)` — it does not call `setDetached(false)`. The sequence "open → detach → close → open" puts the user back into detached mode immediately. This state doesn't survive page reload (plain `useState(false)`), creating an inconsistency: the preference is "remembered" within a session but lost on refresh.

## Spotify button: drag-vs-click conflict and constraint staleness

The snapped button element has both `drag="y"` and `onClick={handleExpand}` on the same `<motion.div>`. Framer Motion suppresses `onClick` during a drag gesture only when the drag exceeds a very small internal threshold. A user who intends to drag vertically but releases quickly will trigger `handleExpand` unexpectedly.

The drag constraints are computed once per render:

```tsx
const maxDragUp = -topPosition;
const maxDragDown = (typeof window !== "undefined" ? window.innerHeight : 800) - SNAPPED_BUTTON_SIZE - topPosition;
```

If the viewport height changes (mobile orientation flip, desktop window resize), these constraints become stale until the component re-renders for another reason. The button could end up outside the visible area or unable to reach the new bottom edge.

## Dashboard unauthenticated redirect

The new guard calls `router.push("/")` on every render while `status === "unauthenticated"`. Next.js App Router deduplicates navigation calls, but wrapping this in a `useEffect` or ref guard would be more explicit and avoid relying on framework deduplication behavior.
