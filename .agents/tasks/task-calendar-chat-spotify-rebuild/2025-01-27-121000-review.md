# Calendar drag fix, smart fetch, AI chat detach, Spotify vertical drag (v2)

This change delivers four UX improvements: calendar events re-render with updated time text on drag via a key that includes `event.start`, fetch scoping narrows the API window per view (Day=1, Week=7, Month=30 days) with `currentDate` in the dependency array, the AI chat panel supports slide-from-right plus a detachable floating mode constrained to the viewport, and the Spotify button gains vertical-only drag with localStorage persistence and a resize listener. A dashboard unauthenticated redirect rounds it out.

**Watch for:** The `fetchCalendarEvents(accessToken, days)` call doesn't pass `currentDate` to the backend, so navigating away from today fetches relative to server-side "now" regardless of which date is displayed (likely). Switching between attached and detached chat mode unmounts `ChatExperience`, which may lose conversation state if messages are held in local component state (possible). The detached panel's initial position is set via CSS `calc()` rather than motion values, so Framer Motion cannot programmatically re-center it after drag (possible — minor UX gap).

**Verdict**: APPROVED

## High-level view

All five issues from the v1 review have been addressed. `currentDate` is now a useEffect dependency, the layout resets `detached` on close, the floating panel uses `dragConstraints={dragConstraintsRef}` with an `inset-0` boundary div, the Spotify button guards clicks with `isDraggingRef`, and a resize listener keeps constraints fresh.

The calendar key mechanism (`${event.id || event.summary}-${event.start}`) forces React to unmount/remount `TimeBlock` when an event is dragged to a new slot, causing it to read the updated ISO strings and display the correct time.

The AI chat panel splits rendering into two branches (detached vs. attached) sharing the same `ChatExperience` child. Switching modes unmounts and remounts the chat, so conversation state depends entirely on `ChatExperience`'s internal persistence — if it holds messages in local state, detaching/attaching loses chat history.

## Calendar fetch scoping and navigation

The useEffect dependency array `[accessToken, view, currentDate]` correctly triggers a refetch when the user navigates. However, the API call is `fetchCalendarEvents(accessToken, days)` where `days` is a view-derived integer (1, 7, or 30). Unless `fetchCalendarEvents` internally uses `currentDate` (which isn't passed), the backend returns events starting from "now" regardless of what date the calendar is displaying.

For a user on the Day view clicking "next" five times (viewing five days from now), the fetch fires with `days=1` — which would return today's events, not the target day's events. The fix is straightforward: pass `currentDate` (or a `startDate`/`endDate` pair) to `fetchCalendarEvents`.

## Chat panel mount lifecycle across modes

The attached panel and detached panel are separate `<motion.div>` subtrees. When `detached` flips from `false` to `true`, React unmounts the attached subtree (including `<ChatExperience>`) and mounts the detached one with a fresh `<ChatExperience>`. If `ChatExperience` holds messages in local state (likely, given there's no visible external store passed in), the conversation vanishes on mode switch.

A shared ref or lifting message state into the parent would preserve continuity. Alternatively, `ChatExperience` could persist messages via a context that survives remounts.
