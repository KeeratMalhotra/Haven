# Landing page polish, Pomodoro scope fix, and task dedup rewrite

This change bundles four fixes: a dark/light mode toggle on the landing page nav, a more prominent Spotify music button, moving the PomodoroTimer to the dashboard layout so it responds from any sub-page, and a complete rewrite of the task load/merge logic to eliminate tripling. The task dedup rewrite is the heaviest lift — it inverts the merge strategy from "localStorage is truth, merge new API tasks in" to "API is truth, overlay local enrichments on top." The Pomodoro and theme toggle changes are mechanical relocations.

**Watch for:** Title-based dedup can silently merge distinct tasks that share the same name (confirmed). The `focusTask` state is declared but never populated, so the timer always opens without a task label (confirmed). The local-only task filter uses a `task-` prefix heuristic that could misfire if API IDs ever start with that prefix (possible).

## High-level view

The task merge rewrite treats the API response as the canonical set of tasks and demotes localStorage to an enrichment layer. Each API task is matched to its local counterpart first by ID, then by title. Unmatched local tasks with `task-`-prefixed IDs survive as "local-only" items. This eliminates the tripling bug where a single Google Task appeared three times due to ID mismatch across page loads. The title-match fallback is the bootstrap mechanism — it fires once to adopt the canonical Google ID, after which future loads match by ID directly. The gap is that tasks sharing a title will collide.

The Pomodoro timer moved from the dashboard index page to the dashboard layout — the correct scope since `chronai-start-focus` is dispatched from the TopBar which lives outside any individual page. However, the event listener only calls `setFocusActive(true)` and never reads a task name from the event, leaving `focusTask` always `undefined`.

The Spotify button styling shifts from a neutral surface color to an emerald-tinted glow with a pulsing indicator dot and larger hit target.

<summary>Issues (4)</summary>

1. **Title collision in dedup** — Two distinct tasks with the same title (e.g., "Meeting prep") will be collapsed into one during the title-match fallback. Add secondary discrimination (due date, notes hash) or at minimum only match when `completed` status agrees.
2. **focusTask never set** — `setFocusTask` is never called; the Pomodoro timer always opens without a task label. Extract the task name from a `CustomEvent.detail` payload if the TopBar has one, or remove the dead state.
3. **local-only prefix heuristic** — The filter `lt.id.startsWith("task-")` determines whether unmatched local tasks survive. If Google Tasks ever returns IDs starting with "task-", those would be incorrectly retained as duplicates. Use a more explicit marker (e.g., a `localOnly: true` flag) or check against a known Google Task ID format.
4. **Multi-tab race on localStorage** — After the API-driven merge, the result is persisted via a `useEffect` on `tasks`. If two tabs load simultaneously, both merge independently and one overwrites the other's enrichments. Not critical for a single-tab app, but a `BroadcastChannel` or storage-event listener would prevent data loss.

## Task merge rewrite and the title-collision gap

The new merge logic iterates over `apiTasks` and for each task attempts an ID lookup in `localById`, falling back to `localByTitle`. When a match is found, local enrichments (status, priority, labels, recurrence, subtasks) are preserved while the API's canonical ID and content fields take precedence.

The title-match fallback is the dedup mechanism that fixes the tripling — the first time a task returns from the API with a real Google ID, it won't match the random `task-<timestamp>-<random>` ID in localStorage, but it will match by title. From that point forward, the canonical ID is persisted and future loads match by ID directly.

The gap: `localByTitle` stores only the first local task with a given title. If a user has two tasks named "Review PRs" (common with recurring tasks), the second one will never match by title and will appear as a "truly new" API task, losing its local enrichments. Conversely, if the API returns two tasks with the same title, both look up the same local task from the map. The `matchedLocalIds` set prevents the local task from being retained as "local-only," but both API tasks receive that single local task's enrichments — one of them incorrectly.

```typescript
// If two API tasks share a title, both get localByTitle.get(title) → same local task
// First call: matchedLocalIds.add("task-abc") ← correct
// Second call: matchedLocalIds.add("task-abc") ← no-op, second API task gets stale enrichments
```

For the 8-task scenario in the user report this works fine — Google Tasks titles are typically unique. But it's a latent bug for anyone with recurring tasks of the same name.

## Dead focusTask state in layout

The layout declares `focusTask` state and passes it to `PomodoroTimer` as `taskName`, but nothing ever calls `setFocusTask`. The event handler only flips `focusActive`:

```typescript
const handleStartFocus = () => {
  setFocusActive(true);
};
```

This isn't a regression — the old `page.tsx` had the same dead state — but it was faithfully moved rather than fixed. If the TopBar ever passes a task name via `CustomEvent.detail`, the layout won't read it without additional wiring.

## Local-only task retention heuristic

Unmatched local tasks survive the merge only if their ID starts with `"task-"`. This distinguishes locally-created tasks (which use `task-<timestamp>-<random>`) from tasks that were once synced with the API (Google-format base64-ish IDs). The heuristic is undocumented and breaks if Google ever issues IDs with that prefix or if the local ID generation scheme changes. A `source: "local"` flag at creation time would be more robust.
