# Omnipresent AI context layer with toasts, banners, and WebSocket notifications (v2)

Sprint 4 adds a global AI context system: `AIContextProvider` wraps the dashboard, exposes `reportAction`, debounces actions into batched calls to `/api/context-suggest` (Gemini-powered), and renders responses as toasts and inline banners. A second WebSocket connection handles server-pushed notifications. All six issues from the v1 review have been addressed: prompt injection mitigated via `system_instruction` separation plus input sanitization, debounce rewritten as a queue/batch, action buttons relabeled to "Got it" instead of misleading AI labels, notification socket auth-failure detection with reconnect backoff, calendar `reportAction` moved after successful API call, and `reportAction` calls moved outside `setTasks` updaters.

**Watch for:** The notification socket "fresh token" read inside `connect()` still reads from the outer `session` closure captured at effect setup, so it cannot truly get a refreshed token after expiry — reconnect attempts will loop with a known-bad token until next-auth triggers a re-render (confirmed). The stale closure fix introduces `tasks` as a dependency of `handleToggleTask` and `handleDeleteTask`, causing these callbacks to recreate on every task array change (possible performance concern for large task lists).

## High-level view

The notification socket now detects auth failure messages and closes/retries after 10 seconds. However, `connect()` reads `session` from the closure captured when the `useEffect` fired, not a truly current value — so reconnecting with the same expired token is the default behavior until next-auth's session polling triggers a re-render and effect re-run.

The stale closure fix moves `reportAction` calls outside `setTasks` updaters, which requires reading `tasks` synchronously via `tasks.find(...)`. This forces `tasks` into the `useCallback` dependency arrays, recreating callback identity on every task mutation and propagating new references to all child components.

## Notification socket token lifecycle

The `useNotificationSocket` hook runs its `useEffect` with `[session, addNotification]` as deps. When the effect fires, it captures `session` in the closure. The inner `connect()` function claims to "get a fresh token at connection time":

```ts
function connect() {
  const currentToken = (session as { accessToken?: string } | null)?.accessToken;
```

This reads the same `session` object captured when the effect last ran. If the socket disconnects due to auth failure and retries after 10 seconds, it will use the same token unless React has re-rendered the component with a new `session` from next-auth in the interim. The `[session, ...]` dependency will eventually trigger a full effect teardown/re-run — but during the 10-second retry window, the reconnection will fail with the same expired token, hit the auth-failure branch again, and retry in another 10 seconds. This creates a retry-with-known-bad-token loop that's bounded but wasteful.

A cleaner pattern: store the token in a ref updated by a separate `useEffect(() => { tokenRef.current = session?.accessToken }, [session])`, then `connect()` always reads `tokenRef.current`.

## Callback identity and child re-renders

The stale closure fix moves `reportAction` calls outside `setTasks` updaters, which requires reading `tasks` synchronously to find the task by ID:

```ts
const handleDeleteTask = useCallback((taskId: string) => {
  const task = tasks.find((t) => t.id === taskId);
  if (task) {
    reportAction("task_deleted", { taskId, title: task.title });
  }
  setTasks((prev) => prev.filter((t) => t.id !== taskId));
  ...
}, [accessToken, reportAction, tasks]);
```

The `tasks` dependency means the callback identity changes on every task mutation. Every `ListRow` and `SortableTaskCard` receives new `onToggle`/`onDelete` props, triggering re-renders of the entire list. The alternative — `tasksRef.current = tasks` maintained by a separate effect, read inside the callback — would keep the callback identity stable while still avoiding the stale-closure bug.
