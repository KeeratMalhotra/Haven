# Omnipresent AI context layer with toasts, banners, and WebSocket notifications

Sprint 4 adds a global AI awareness system that observes user actions across the dashboard and delivers contextual suggestions. An `AIContextProvider` wraps the dashboard, exposes `reportAction` to child pages, debounces calls to a new backend `/api/context-suggest` endpoint (Gemini-powered), and feeds responses into a toast/banner UI. A second WebSocket connection listens for server-pushed notifications. Separately, GreetingHero and VoiceMode were restyled to drop gradients and Three.js in favor of matte CSS-variable theming.

**Watch for:** User-supplied `action_data` is interpolated into the Gemini prompt with only length truncation â€” no structural sanitization against prompt injection (confirmed). The `reportAction` debounce uses last-write-wins, so rapid consecutive actions (drag then create within 500ms) silently drop the first event (confirmed). The notification WebSocket authenticates via "ping" type â€” auth failures are invisible to the hook, which will sit connected but unauthenticated with no retry (likely). Action buttons rendered from AI suggestions are no-ops that only dismiss the banner (confirmed).

## High-level view

The AI context layer is a React context + debounced API call. Components call `reportAction` with a string type and data object; a 500ms debounce coalesces rapid actions into a single Gemini request â€” but the coalescing is destructive (last-write-wins, not batched), so the AI misses events in rapid sequences.

The backend `/api/context-suggest` validates auth, truncates inputs to length limits, constructs a single-turn Gemini prompt, and returns a structured suggestion or null. User data is interpolated directly into the prompt instruction text rather than separated into a user-role message, creating a prompt injection surface. The blast radius is limited to misleading toast text, not data exfiltration.

The notification socket opens a second WebSocket to `/ws`, sends a "ping" with auth token. The backend's handler does register the connection with `ConnectionManager` before routing the message type, so pushed notifications should arrive. The gap: if auth fails (expired token), the hook receives an error JSON it ignores and never retries auth, leaving the socket connected but unregistered.

The suggestion UI feeds both `AIToast` (floating, auto-dismiss 8s) and `AISuggestionBanner` (inline, first undismissed). Action buttons in both components route to `dismissSuggestion` â€” no actual functionality (reschedule, prioritize) is wired, making them misleading to users.

## Prompt construction and injection surface

The `/api/context-suggest` endpoint builds a Gemini prompt by interpolating `action_type`, `action_data_str`, and `context_str` directly into the instruction text:

```python
f"Action type: {action_type}\n"
f"Action data: ```{action_data_str}```\n"
f"Context: ```{context_str}```\n\n"
```

The inputs are length-truncated (100 and 500 chars) but not structurally sanitized. Since `action_data` comes from client-side JavaScript, a compromised or malicious client could craft payloads that manipulate the model's behavior. The triple-backtick wrapping provides no security boundary for LLMs.

The blast radius is limited: the worst case is the model returning a misleading suggestion string displayed in a toast â€” not a data exfiltration vector. Still, separating instruction from user data (using system/user message roles in the Gemini API) would close this gap.

## Debounce semantics â€” last-write-wins drops actions

`AIContextProvider.reportAction` sets `pendingActionRef.current` to the latest action and resets the debounce timer on each call:

```ts
pendingActionRef.current = { actionType, actionData };
if (debounceRef.current) clearTimeout(debounceRef.current);
debounceRef.current = setTimeout(async () => { ... }, 500);
```

Rapid successive actions (drag-then-create, or multiple drags in quick succession) only send the *last* one to the backend. The stated goal is "the AI notices everything" â€” this pattern contradicts that. A queue that batches actions over 500ms and sends all of them would preserve the full action stream.

## Notification socket auth failure invisibility

The hook authenticates via:

```ts
ws.send(JSON.stringify({ type: "ping", auth_token: accessToken }));
```

The backend validates the token and registers the connection before routing the message type. A "ping" type doesn't match any route, falls through silently, and the connection stays alive â€” so push notifications *will* arrive for valid tokens.

The failure case: if the token is expired or malformed, the backend sends `{type: "error", content: "Authentication failed..."}`. The hook's `onmessage` handler only processes `type === "notification"` messages, so this error is silently ignored. The socket stays open, unregistered with `ConnectionManager`, and never receives pushed notifications. There's no retry-auth logic, no user-visible indication, and no exponential backoff for this scenario. Token expiry during long sessions would trigger this silently.

## Action buttons route nowhere

Both `AISuggestionBanner` and `AIToast` render action buttons sourced from Gemini's response:

```ts
actions={activeSuggestion.actions?.map((a) => ({
  label: a.label,
  onClick: () => dismissSuggestion(activeSuggestion.id),
}))}
```

Every button dismisses the suggestion. No routing to actual functionality exists. If Gemini returns `{label: "Reschedule", action: "reschedule_tasks"}`, clicking "Reschedule" just removes the banner. This erodes user trust â€” either suppress action buttons until handlers exist, or wire them to the chat/agent system.

## Calendar reportAction ordering

In `handleCreateEvent`, `reportAction("event_created", ...)` fires before the `createCalendarEvent` API call:

```ts
reportAction("event_created", { summary, startTime, duration });
try {
  const created = await createCalendarEvent(accessToken, { ... });
```

If the API call fails (network error, auth expired, quota), the AI was told an event was created that doesn't exist. The suggestion it generates may reference a phantom event. The tasks page doesn't have this problem â€” `handleCreateTask` calls `reportAction` after optimistic local addition, which is at least consistent with what the user sees even if the API fails later.
