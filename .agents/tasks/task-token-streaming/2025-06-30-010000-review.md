# Token-by-token streaming for WebSocket chat

The change adds real-time token streaming from Gemini through the existing WebSocket transport. Instead of buffering a full response and sending it in one frame, the backend now streams `text_chunk` frames as the model generates tokens, followed by a `text_end` sentinel. The frontend accumulates chunks into a growing message and skips its client-side typewriter effect for externally-streamed content. The approach routes streaming only through single-agent dispatches; multi-agent consolidation and direct responses fall back to the original full-frame path.

**Watch for:** Chunk iteration is batched in a thread rather than yielded incrementally, defeating true token-by-token delivery (confirmed). The `_stream_callback` is set as mutable shared state on a singleton agent, which creates a race condition under concurrent WebSocket connections (confirmed). If the model times out or errors after `_streamed` is set True, the client receives `text_end` with no prior chunks, creating a ghost message (likely).

## High-level view

The streaming pipeline threads through three layers: `AgentBase._generate_with_streaming` collects all chunks in a background thread before forwarding them to the callback, `OrchestratorAgent.execute_streaming` wires the callback from the WebSocket handler to the specialist agent, and `main.py` frames the chunks as JSON over the wire. The batching in `_generate_with_streaming` means the user sees all tokens arrive in a burst after generation completes rather than incrementally as the model produces them.

The concurrency model is the central risk. `_stream_callback` is an instance attribute on a singleton agent object. If two users send messages simultaneously, the second connection's callback overwrites the first, and chunks from both responses interleave or route to the wrong client. The `try/finally` clearing of the callback doesn't help when the race happens before the first call enters `generate()`.

On the frontend, if the backend encounters an error mid-stream and never sends `text_end`, the message stays in `streaming: true` state indefinitely with no client-side timeout to finalize orphaned streaming messages.

<details>
<summary>Issues (4)</summary>

1. **Batched chunk iteration defeats streaming** — `_iter_chunks()` collects all chunks into a list before returning, so the callback fires in a burst after the model finishes. Yield chunks one at a time from the thread (e.g., via `asyncio.Queue`) to deliver true incremental streaming.
2. **Singleton callback race under concurrency** — `agent._stream_callback` is set on a shared singleton instance. Two concurrent requests to the same specialist agent will overwrite each other's callback. Use a per-request context (pass the callback through `task` dict or use `contextvars`) instead of mutating agent state.
3. **Empty/failed stream produces ghost message** — If `generate()` returns the fallback (empty string) due to timeout/error while `_streamed` is still True, the client receives `text_end` with no prior chunks, creating a message entry that never gets content. Send a `text` frame with the fallback content instead of `text_end` when no chunks were actually emitted.
4. **`generate_stream()` is dead code** — The async generator method in `AgentBase` is never called by the orchestrator or handler. The actual streaming path uses `_stream_callback` + `_generate_with_streaming`. Either remove it or wire it up.

## Batched chunk delivery in `_generate_with_streaming`

The streaming helper in `base.py` calls `asyncio.to_thread(_iter_chunks)` where `_iter_chunks` iterates the entire Gemini response iterator and collects results into a list before returning. Only after the list is fully built does the code loop through it and call `await self._stream_callback(chunk_text)` for each entry. This means the user sees nothing during generation and then receives all tokens in rapid succession once the model finishes — functionally identical to the non-streaming path from a UX perspective.

```python
def _iter_chunks():
    chunks = []
    for chunk in response_iter:
        text = chunk.text if hasattr(chunk, "text") else ""
        if text:
            chunks.append(text)
    return chunks

chunks = await asyncio.to_thread(_iter_chunks)  # blocks until all done
for chunk_text in chunks:
    await self._stream_callback(chunk_text)  # fires in burst
```

To deliver true incremental streaming, each chunk needs to cross the thread boundary as it arrives. An `asyncio.Queue` with a producer thread and an async consumer loop is the standard pattern for bridging blocking iterators into async code.

## Shared mutable callback on singleton agents

The orchestrator sets `agent._stream_callback = send_chunk` before calling `agent.execute()` and clears it in a `finally` block. Every specialist agent is a singleton registered in `AgentRegistry`. If two WebSocket connections concurrently route to the same specialist (e.g., both ask the scheduler), the second assignment overwrites the first. Chunks from the first user's generation will be sent to the second user's WebSocket, or lost entirely.

```python
agent._stream_callback = send_chunk
try:
    result = await agent.execute(...)
finally:
    agent._stream_callback = None
```

The fix is to avoid mutating shared agent state: pass the callback through the `task` dict, use `contextvars`, or create per-request agent instances for the streaming path.

## Frontend orphaned streaming state

If the backend encounters an error mid-stream and never sends `text_end`, the message stays in `streaming: true` state indefinitely. The `MessageBubble` will continue showing the streaming caret CSS class. There's no client-side timeout to finalize orphaned streaming messages. A 30-second watchdog timer that flips `streaming` to false if no new chunk arrives would prevent permanently-stuck UI state.

## Test coverage gaps

The test file covers the happy path for streaming (chunks arrive in order, `text_end` follows) and the non-streamed fallback.

Not tested: the actual `_generate_with_streaming` method against a mock model, concurrent streaming to the same agent, error/timeout mid-stream behavior, the interaction between `_stream_callback` and `generate()`'s fallback path, or the frontend `MessageBubble` streaming detection logic. The `generate_stream()` async generator has no tests and no callers.
