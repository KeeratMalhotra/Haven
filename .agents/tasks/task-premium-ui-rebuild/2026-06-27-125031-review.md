# Premium UI/UX rebuild: Notion-inspired productivity workspace

A ground-up frontend redesign replaces the previous single-page chat-centric dashboard with a multi-route productivity workspace. The old architecture (ambient canvas background, SideDock with slide-in drawers, central ChatExperience) gives way to a persistent sidebar + app shell layout with dedicated routes for Tasks, Calendar, Habits, and Settings, plus a design system of reusable UI primitives. The design language shifts from neon magenta/cyan with glassmorphism to Notion-style restraint: indigo/violet accents, subtle surfaces, and Inter as primary typeface with system font fallbacks.

**Watch for:** Connection status indicator is permanently stuck on "connecting" and will never show the green dot (confirmed). AI chat suggestion chips are inert `<span>` elements with no click handler (confirmed). Drag-and-drop across kanban columns relies on task-to-task overlap rather than column containers, so dropping on empty columns does nothing (likely). Calendar page creates events with no timezone handling, which will produce incorrect time block positioning for non-UTC users (likely).

## High-level view

The new dashboard layout component hardcodes the WebSocket connection status to `"connecting"` without ever subscribing to the actual socket, meaning the TopBar's status dot always shows amber/pulse regardless of real connectivity. This is visible on every dashboard route.

The AI integration shifted from a central full-screen chat to a floating action button + slide-in side panel wrapping ChatExperience. Suggestion chips at the panel bottom are rendered as non-interactive spans. The command palette lost its ability to compose AI chat messages ("Add event...", "Add task..."), removing the product's stated AI differentiator from quick-access workflows.

Tasks integrates `@dnd-kit` for kanban drag-and-drop but only resolves drops by finding another task at the destination position. Empty columns have no droppable surface. Within columns, no `arrayMove` logic handles reordering despite `SortableContext` being present.

Calendar implements month/week/day views with date-fns arithmetic. Event time blocks are positioned using `differenceInMinutes` against a local 6AM baseline without normalizing UTC timestamps from the API, so non-UTC users will see shifted blocks.

Settings writes AI preferences (tone, suggestions toggle, proactive notifications) to localStorage, but no component outside Settings reads them. The toggles are functionally inert.

Several components hard-code `bg-base-950`, `text-white`, and raw `rgba(...)` values that bypass the CSS variable theme system. These will render incorrectly when the user switches to light mode via the theme toggle in Settings or Sidebar.

<details>
<summary>Details</summary>

## Connection status indicator never resolves

`frontend/app/dashboard/layout.tsx` introduces:

```tsx
const [connection] = useState<ConnectionState>("connecting");
```

This state never updates. The `useChatSocket` hook that manages WebSocket lifecycle is imported only for the `ConnectionState` type, never instantiated. The previous dashboard page passed an `onConnectionChange` callback to `ChatExperience` and wired it to the TopBar; that plumbing was removed. The connection dot is visible on every dashboard route since it lives in the shared layout.

## Suggestion chips are non-interactive

`AIChatPanel` renders suggestion chips as:

```tsx
<span className="... cursor-default">
  {suggestion}
</span>
```

No `onClick`, no `ref` forwarding to `ChatExperience`, no mechanism to inject a message into the chat. `cursor-default` explicitly signals "not clickable."

## Kanban drag-and-drop cross-column gap

The `handleDragEnd` logic resolves the destination by looking up which task the dragged card was dropped on:

```tsx
const overTask = tasks.find((t) => t.id === over.id);
if (overTask && overTask.status !== activeTask.status) {
  // move to overTask's column
}
```

If a column is empty, there is no `over` target. `@dnd-kit` supports `useDroppable` for container-level drop zones, but none are registered. Within a column, `SortableContext` + `verticalListSortingStrategy` is provided but no `arrayMove` logic handles intra-column reordering, so drags within a column snap back.

## Calendar timezone assumptions

`TimeBlock` calculates vertical position via:

```tsx
const startMinutes = differenceInMinutes(startDate, setHours(setMinutes(dayStart, 0), 6));
```

If the API returns ISO strings with `Z` suffix, `parseISO` gives UTC, and `differenceInMinutes` against a local `dayStart` produces wrong offsets. Events will appear shifted for non-UTC users. The backend (Google Calendar API) returns timezone-aware strings, but the front-end doesn't normalize before computing pixel positions.

## Command palette loses AI composition

The previous `CommandPalette` accepted `onSendChat` and used it to compose natural-language commands ("Add event: ", "Add task: ", "Check in habit"). The rewrite removes this prop:

```tsx
interface CommandPaletteProps {
  onFocusMode?: () => void;
  onOpenChat?: () => void;
}
```

"Create Task" and "Create Event" actions now simply navigate to pages. "Ask AI" opens the chat panel but doesn't seed it with a prompt. Given the user's goal of "AI integrated everywhere," this removes the fastest AI interaction path.

## Light mode theme breakage

Components that bypass CSS variables and will render incorrectly in light mode:

- `FocusMode`: `bg-base-950` background, `text-white` labels
- Landing page: multiple `text-white` elements over transparent backgrounds
- `Sidebar` collapsed tooltip: `bg-gray-900 text-white`
- `FloatingOrb` on landing page: hard-coded `rgba(...)` opacity values that assume dark context

Dark mode (default) is fine. These only manifest when the user toggles theme in Settings.

## Settings AI preferences write-only

Settings stores three AI-related values in localStorage (`chronai-ai-tone`, `chronai-ai-suggestions`, `chronai-proactive-notifs`). No component outside Settings reads them. `ChatExperience` (unchanged, out of scope) doesn't consume tone preferences. The toggles exist as UI elements but don't affect behavior. This is acceptable as scaffolding for future integration, but the Settings page implies they're active.
