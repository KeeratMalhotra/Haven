# Premium UI polish: fixes validated, remaining edges (v2)

All six issues from the v1 review have been addressed. The profile picture enforces a 2 MB cap with `QuotaExceededError` handling. The toast system reuses a single module-level `AudioContext`. The `chronai-start-focus` event listener is wired up in `dashboard/page.tsx`. Spotify play/pause and volume are documented as visual-only. The detached chat panel uses `max()` clamping and falls back to full-screen on viewports under 640px.

**Watch for:** Spotify expanded mode mounts a second iframe that will restart playback (confirmed). The detached panel's responsive gate uses a one-time `window.innerWidth` check that won't re-evaluate on resize (confirmed). `typeof window` guards in render produce hydration mismatches with SSR (likely).

## High-level view

The Spotify widget's dual-iframe architecture keeps music alive when snapped, but expanding mounts a fresh `<iframe src={embedUrl}>` that triggers a new embed initialization — the user hears a restart or brief silence when switching from snapped to expanded. A single always-present iframe repositioned between the offscreen slot and the visible card would eliminate this.

The detached chat panel's small-screen fallback checks `window.innerWidth < 640` at render time and branches to a completely different JSX tree. If the user resizes their browser from wide to narrow (or rotates a tablet), the panel stays in the stale mode until the component re-mounts.

Several components (`SpotifyMiniPlayer`, `AIChatPanel`) use `typeof window !== "undefined" ? window.innerWidth : 400` directly in inline styles. During SSR the server renders with the fallback value; the client immediately hydrates with the real viewport, producing a React hydration mismatch warning and potential layout flicker on first paint.

<summary>Issues (3)</summary>

1. **Dual iframe restarts playback on expand** — When toggling from snapped to expanded, a second `<iframe>` with the same `src` mounts and reinitializes the Spotify embed. Use a single always-present iframe and reparent or reposition it rather than conditionally rendering two.
2. **Detached panel responsive gate is static** — `window.innerWidth < 640` is evaluated once at render. Add a resize/orientation listener or use a CSS media query (e.g., `useMediaQuery` hook) so the panel switches modes dynamically.
3. **SSR hydration mismatch from inline window dimension checks** — Inline `typeof window !== "undefined" ? window.innerWidth : 400` produces different server/client values. Use a `useEffect` + state pattern or `suppressHydrationWarning` on affected elements.

## Spotify dual-iframe restart on expand

The offscreen container renders one `<iframe src={embedUrl}>` that keeps music playing while snapped. When the user expands:

```tsx
{/* Always-mounted iframe container */}
{playerState.snapped && (
  <div style={{ left: -9999, opacity: 0, pointerEvents: "none" }}>
    <iframe src={embedUrl} ... />
  </div>
)}

{/* Expanded card state */}
{!playerState.snapped && (
  <motion.div ...>
    ...
    <iframe src={embedUrl} ... />
  </motion.div>
)}
```

The conditional rendering means the snapped iframe unmounts when expanding and a new one mounts inside the card. The Spotify embed reinitializes on mount — the user hears playback restart. This defeats the original "never unmount" design goal. A single iframe rendered unconditionally and repositioned (via `ref` + portal or absolute positioning switch) would keep the same embed instance alive across both visual states.

## Detached panel viewport gate evaluated once

```tsx
if (detached) {
  const isSmallScreen = typeof window !== "undefined" && window.innerWidth < 640;
  if (isSmallScreen) {
    return ( /* full-screen fallback */ );
  }
  return ( /* floating window */ );
}
```

There's no state or subscription tracking viewport changes. A user who opens the panel in landscape then rotates to portrait sees the floating window clip offscreen. A `useMediaQuery("(min-width: 640px)")` hook or a resize effect updating state would make this reactive.

## SSR hydration mismatches

`SpotifyMiniPlayer`:

```tsx
const [viewportHeight, setViewportHeight] = useState<number>(
  typeof window !== "undefined" ? window.innerHeight : 800
);
```

`AIChatPanel`:

```tsx
style={{
  width: Math.min(400, typeof window !== "undefined" ? window.innerWidth - 32 : 400),
  ...
}}
```

On the server, `window` is undefined so the fallback fires. On the client, the real value fires immediately in the same render pass. Since both components are `"use client"` and Next.js 14 renders client components on the server for initial HTML, this produces hydration warnings. The fix: initialize state to the fallback, then set the real value in a `useEffect` after mount.
