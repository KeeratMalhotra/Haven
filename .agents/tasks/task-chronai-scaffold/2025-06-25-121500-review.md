# Full-stack AI productivity companion with particle UI entity (v2)

All seven issues from the first review have been addressed. Auth tokens propagate from NextAuth through the session to the WebSocket. Conversation history is per-connection. AudioAnalyzer drives shader uniforms with a gesture-gated initialization. Frontend Dockerfile exists as a proper multi-stage build. Voice agent uses a dedicated `GCP_API_KEY` with fallback. Shared schemas are imported and used for validation in the WebSocket handler.

**Watch for:** The backend `verify_google_token` calls `id_token.verify_oauth2_token` which validates JWT *ID tokens*, but the frontend sends the OAuth *access token* (an opaque string, not a JWT) — these are different credentials and the verification will fail at runtime (confirmed). The AudioAnalyzer is initialized on click but has no audio source connected to it, so `getAverageFrequency()` will always return 0 and the idle simulation branch will remain active (confirmed). The frontend Dockerfile bind-mounts source over the built image in docker-compose, defeating the multi-stage build in development (likely).

## High-level view

The auth pipeline now persists `account.access_token` in the JWT callback and exposes it to the client via session. ChatPanel reads it and sends it on every WebSocket message. The gap is that the backend validates it with `verify_oauth2_token` — a function designed for ID tokens (JWTs issued by Google's OIDC provider) — not access tokens. Access tokens are opaque strings that can't be locally verified this way. The pipeline will throw a ValueError on every authenticated message.

The AudioAnalyzer now instantiates in ParticleSystem and initializes its AudioContext on first click. However, `init()` only creates the context and analyser node — it never calls `connectAudioElement` to wire an actual audio source. Without a source node feeding the analyser, frequency data will always be zero. The code path branches on `avgFrequency > 0.01`, which will never be true, so the idle simulation remains permanently active.

The shared schemas `sys.path` import works in local development but will break in the docker-compose container. The path resolves to `Path("/app").parent.parent / "shared"` which lands at `/shared` — a path that doesn't exist in the container. The volumes section doesn't mount the shared package.

## Access token sent where ID token expected

The NextAuth JWT callback now persists `account.access_token` and the session exposes it to the client. ChatPanel sends it on every WebSocket message. On the backend, `auth.py` does:

```python
id_info = id_token.verify_oauth2_token(
    token,
    google_requests.Request(),
    settings.GOOGLE_CLIENT_ID,
)
```

`verify_oauth2_token` from `google.oauth2.id_token` expects a signed JWT issued by Google's identity platform. An OAuth access token is a different artifact — typically an opaque string like `ya29.a0...`. Passing it to this function will raise `ValueError("Wrong number of segments in token")` or a signature verification failure.

The architectural tension: the MCP servers need the access token to call Google APIs (Calendar, Tasks), but the backend wants to verify the user's identity. The typical resolution is to send the ID token for authentication and the access token for API calls, or to verify the access token via Google's tokeninfo endpoint (`https://oauth2.googleapis.com/tokeninfo?access_token=...`).

## AudioAnalyzer wired but sourceless

The fix gates AudioContext initialization behind a user gesture:

```typescript
const initAudio = useCallback(async () => {
  if (audioInitialized.current) return;
  audioInitialized.current = true;
  try {
    await audioAnalyzer.current.init();
  } catch {
    audioInitialized.current = false;
  }
}, []);
```

`init()` creates `AudioContext` and `AnalyserNode` but `AnalyserNode.getByteFrequencyData` only produces non-zero output when a source node is connected. The AudioAnalyzer class has `connectAudioElement(audioElement: HTMLAudioElement)` for this purpose, but nothing calls it. The voice agent's TTS audio plays via `playAudioBase64` in `voice.ts` (which creates a temporary `Audio` element), but that element is never passed to the analyzer.

## Shared schemas path resolution in Docker

```python
_shared_path = str(Path(__file__).resolve().parent.parent.parent / "shared")
if _shared_path not in sys.path:
    sys.path.insert(0, str(Path(__file__).resolve().parent.parent.parent))
```

In docker-compose, the backend's working directory is `/app`. `Path("/app/app/main.py").parent.parent.parent` resolves to `/`, so `from shared.schemas import ...` looks for `/shared/schemas.py`. The compose file mounts `./backend:/app` and `./mcp-servers:/mcp-servers` but never mounts `./shared` anywhere the path resolution can find it.
