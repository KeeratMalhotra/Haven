# Full-stack AI productivity companion with particle UI entity

ChronAI is a greenfield full-stack application: a Next.js 15 frontend with a custom Three.js particle system driven by GLSL shaders, backed by a Python FastAPI server that uses Gemini 2.5 Flash for multi-agent orchestration, with MCP servers bridging to Google Calendar and Tasks APIs. The architecture is clean — frontend communicates over WebSocket with a well-defined JSON protocol, agents register and route through an orchestrator, and MCP servers run as stdio subprocesses discovered at startup.

**Watch for:** The auth token from NextAuth.js is never actually passed to the WebSocket (confirmed — `auth_token` is always `""`), meaning the entire agent pipeline that depends on Google API access tokens will fail at runtime. The orchestrator holds conversation history as instance state on a single object shared across all WebSocket connections (confirmed — single OrchestratorAgent registered globally), creating cross-user message leakage. The AudioAnalyzer is fully implemented but never wired into the ParticleSystem render loop (confirmed), so audio reactivity is simulated rather than real.

## High-level view

The particle system has real GLSL shaders implementing noise-based idle animation, distance-based mouse attraction, and frequency-band audio displacement. The gap is that audio reactivity uses hardcoded sine-wave simulation because the AudioAnalyzer class is never connected to the shader uniforms — it exists as dead code.

The backend agent architecture uses a single OrchestratorAgent instance (registered at startup, retrieved from the global registry by all WebSocket handlers) that accumulates `conversation_history` from every connected client without isolation. This is both a privacy leak and an unbounded memory growth path.

The WebSocket protocol matches the specified contract (send: `{type, content, auth_token}`, receive: `{type, content, agent}`), but the frontend hardcodes `auth_token: ""` in every message. NextAuth.js is configured for Google OAuth sign-in and route protection, but the OAuth access token (needed by Google APIs) is never persisted in the JWT callback — only `token.sub` propagates. The entire MCP tool pipeline is unreachable at runtime.

MCP servers implement all required tools with proper JSON schemas. The `find_free_slots` implementation strips timezone info for comparison (`replace(tzinfo=None)`) which could produce incorrect results if events span timezone boundaries — acceptable for v1 but worth noting.

<summary>Details</summary>

## Auth token gap between NextAuth and WebSocket

The NextAuth.js configuration in `route.ts` defines a `jwt` callback that simply returns the token unchanged — it never captures `account.access_token` from the OAuth flow. The `session` callback only sets `session.user.id = token.sub`. The Google OAuth access token is discarded after sign-in.

In `ChatPanel.tsx`, every call to `wsRef.current.send()` passes `auth_token: ""`:

```typescript
wsRef.current.send({
  type: "chat",
  content: input.trim(),
  auth_token: "",  // Always empty
});
```

The backend checks `if auth_token:` before verification, so with an empty string it skips entirely and sets `user = None`. Agents guard MCP calls on `if self.mcp_client and auth_token`, meaning the entire productivity pipeline — creating tasks, finding calendar slots, generating notifications — is unreachable.

## Cross-connection state sharing in OrchestratorAgent

The orchestrator maintains `self.conversation_history: list[dict] = []` as instance state. There's only one OrchestratorAgent instance (created at startup, retrieved from the registry by all WebSocket handlers). Every connected client's messages append to the same list, and the LLM sees the combined history of all users. History also grows unbounded — there's no truncation or sliding window.

## AudioAnalyzer dead code

`AudioAnalyzer.ts` is a complete Web Audio API wrapper that extracts frequency data and splits into bass/treble bands. `ParticleSystem.tsx` never imports it. The `useFrame` loop sets audio uniforms to hardcoded sine waves:

```typescript
const idlePulse = Math.sin(time * 2) * 0.02 + 0.02;
material.uniforms.uAudioFrequency.value = idlePulse;
material.uniforms.uAudioBass.value = Math.sin(time * 1.2) * 0.03 + 0.03;
material.uniforms.uAudioTreble.value = Math.sin(time * 3.5) * 0.01 + 0.01;
```

The architecture supports real audio (uniforms exist, analyzer produces the right data shape), but the connection is missing.

## Voice agent API key mismatch

`VoiceAgent._synthesize_speech` calls Google Cloud TTS at `texttospeech.googleapis.com/v1/text:synthesize` using `params={"key": settings.GEMINI_API_KEY}`. The Gemini API key (for `generativelanguage.googleapis.com`) is a distinct credential from a GCP API key authorized for Cloud TTS. TTS calls will likely return 403 unless the user stores a multi-service GCP key in this env var.

## Missing frontend Dockerfile

`docker-compose.yml` declares `build: { context: ./frontend, dockerfile: Dockerfile }` for the frontend service, but no `frontend/Dockerfile` exists in the codebase. The backend Dockerfile exists and is correctly structured.

## Shared schemas not integrated

`shared/schemas.py` defines `WebSocketMessage` and `AgentResponse` as Pydantic models matching the protocol spec. `backend/app/main.py` never imports them — it manually does `data = json.loads(raw_data)` and accesses keys directly. The shared package also lacks a `pyproject.toml` or path configuration that would make it importable from the backend without `sys.path` manipulation.
