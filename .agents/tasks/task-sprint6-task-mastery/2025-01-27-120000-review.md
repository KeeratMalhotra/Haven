# Sprint 6 "Task Mastery" — five features across task management, calendar planning, and AI automation

This sprint adds recurring tasks with auto-regeneration, a labels/tags system with filtering, right-click context menus, bulk multi-select actions, a drag-to-calendar planner page, an AI Auto-Pilot mode (ask permission + full auto), and a template library with AI generation. The backend gains three new routers (`autopilot`, `templates`, and extended task CRUD in `main.py`) while the frontend grows seven new component files and significantly extends the tasks and calendar pages. State management is localStorage-first with optional sync to Google Tasks via the backend MCP bridge.

**Watch for:** Auth token passed in request bodies instead of headers means tokens appear in server logs and request payloads (confirmed). The autopilot `plan_id` is not stored server-side, so the execute endpoint accepts arbitrary action lists from any client without validating them against a previously generated plan (confirmed). Template generation injects user-supplied text into a prompt with only a "treat as opaque data" instruction and no structural separation via system messages (confirmed).

---

## High-level view

The calendar page was rewritten to use `@dnd-kit` for drag-repositioning of events, with a new overlap-group algorithm and conditional compact/medium sizing that fixes the card-fitting bug. Time blocks now use a left accent stripe, clamped height, and `overflow-hidden` to prevent spill, with readable 28px minimum for short events.

The tasks page gained the most surface area: recurring tasks auto-regenerate a new instance on completion, labels are persisted in localStorage via a `useLabels` hook, and the kanban/list views now support checkbox multi-select with a floating bulk-action bar. All local task mutations are opportunistically synced to Google Tasks but failures are swallowed silently — users won't know their changes aren't persisted remotely.

A new `/dashboard/planner` page provides a split-view (task list left, day calendar right) where tasks can be dragged onto time slots. Dropping triggers a confirmation modal with duration selection, then creates a linked calendar event.

Auto-Pilot exposes two modes controlled by a localStorage toggle. "Ask Permission" generates a plan and renders toggleable action cards before execution; "Full Auto" shows a warning, then generates and immediately executes. The backend plan endpoint is stateless — it generates a `plan_id` UUID but never stores it, so the execute endpoint trusts whatever actions the client sends.

The template library ships four built-in workflow templates and a custom-generation endpoint that calls Gemini with the user's goal description. The prompt injection mitigation uses a "treat as opaque data" comment in a single prompt string rather than the system-message separation pattern used elsewhere in the codebase.

<summary>Issues (9)</summary>

1. **Auth tokens in request bodies** — `auth_token` is passed as a JSON field in POST/PATCH bodies (autopilot, templates, tasks). Move to `Authorization` header to avoid token leakage in server access logs and error payloads.
2. **Stateless plan_id allows arbitrary execution** — The execute endpoint accepts any action list; `plan_id` is decorative. Store generated plans server-side (e.g. Firestore with TTL) and validate on execute, or at minimum verify the action structure matches what was generated.
3. **Template prompt injection surface** — User goal text is interpolated into a single prompt with only a prose instruction to ignore embedded commands. Use the `system_instruction` parameter (as done in `context-suggest`) and wrap user input in explicit delimiters.
4. **Autopilot prompt includes unsanitized external data** — Task titles and calendar event summaries from Google APIs are serialized into the prompt without sanitization. A malicious calendar event summary could inject instructions.
5. **Silent failure on task sync** — Task create/delete/update failures in Google Tasks are `.catch(() => {})` with no user feedback. Users won't know their changes aren't persisted remotely.
6. **Custom recurrence `days` field is ignored** — The `RecurrenceSelector` UI lets users pick specific weekdays, but `getNextDueDate` only uses `interval` and ignores the `days` array. The day-picker is non-functional.
7. **Bulk action bar dropdowns are hover-only** — Submenus for "Move to", "Priority", and "Label" use CSS `group-hover:flex`, making them inaccessible on touch devices.
8. **LocalStorage as source of truth diverges from API** — Labels, priority, recurrence, and other local mutations won't round-trip through the backend. Multi-device users will see inconsistent state.
9. **No test coverage for any Sprint 6 code** — No new test files appear in this diff. Overlap algorithm, recurrence logic, and template parsing are untested.

## Calendar card-fitting rewrite and drag-to-reposition

The `TimeBlock` component now uses `@dnd-kit/core` for drag-repositioning. Height is clamped with `Math.max(28, ...)` and `Math.min(height, gridHeight - top)` to prevent overflow past the grid bottom. Short events (< 40px) get compact padding; taller ones get standard spacing. The `EventPill` in month view switched from `<button>` to `<div role="button">` to avoid nested-button hydration errors when rendered inside clickable day cells — a non-obvious fix for a React SSR constraint.

An overlap detection system (`computeOverlaps`) groups concurrent events by sweep-line and assigns horizontal slices. When a drag-to-reposition fails at the API level, the event snaps back because local state isn't updated, but there's no explicit error feedback to the user — the `setError` call exists but depends on an error banner being rendered in the current viewport scroll position. (confirmed)

## Recurring tasks: custom days field is dead code

When a task is toggled to "done" and carries a `recurrence` config, the toggle handler creates a new `LocalTask` with a due date computed by `getNextDueDate`:

```ts
case "daily": return addDays(baseDate, 1)
case "weekly": return addWeeks(baseDate, 1)
case "monthly": return addMonths(baseDate, 1)
case "custom": return addDays(baseDate, recurrence.interval || 1)
```

The "custom" mode accepts a `days` field (specific weekdays) but `getNextDueDate` ignores it — it just adds `interval` days regardless. The `RecurrenceSelector` renders a full weekday picker (`Sun Mon Tue...`) that persists the selection but has zero effect on scheduling. (confirmed)

## Auto-Pilot stateless plan and execution trust

The `/api/autopilot/plan` endpoint returns a `plan_id` (random UUID), `actions`, and `summary`. **Nothing is stored server-side.** The `/api/autopilot/execute` endpoint accepts `plan_id`, `auth_token`, and `actions` from the client and blindly iterates through them calling MCP tools.

This means a client can fabricate actions that were never in the generated plan. The `plan_id` serves no verification purpose. The practical risk is limited by the fact that execution requires a valid `auth_token` (user can only modify their own calendar), but it undermines the "AI generates, user approves" contract of Ask Permission mode. A crafted request could create arbitrary events or move existing ones without plan-generation guardrails.

## Template and autopilot prompt injection asymmetry

The `context-suggest` endpoint uses the proper structural pattern:

```python
model = genai.GenerativeModel(
    settings.GEMINI_MODEL,
    system_instruction=system_instruction,
)
response = await model.generate_content_async(user_data_message)
```

But templates (`/api/templates/generate`) and autopilot (`/api/autopilot/plan`) concatenate user/external input directly into a single prompt string. For templates, the user's goal description is wrapped only in triple-backtick fences with a prose instruction to "treat as opaque data". For autopilot, task titles and calendar event summaries from Google APIs — which could be attacker-controlled — are serialized into the prompt without any sanitization. Both should adopt the `system_instruction` separation pattern.

## Bulk action bar: hover-only dropdowns

The floating bulk-action bar's sub-menus ("Move to", "Priority", "Label") use `hidden group-hover:flex` CSS. On mobile/touch devices, `:hover` doesn't trigger on tap, making these dropdowns inaccessible. Adding a click-toggle state or using a different interaction pattern (e.g., a popover) would fix touch support. (confirmed)

## Planner page: one-way task-event linking

When a task is dropped onto a calendar slot and confirmed, the planner stores the returned event ID as `linkedEventId` on the local task. This linkage is one-way — deleting the calendar event doesn't clear the task's `linkedEventId`, and there's no mechanism to navigate from the event back to the task. The confirmation modal is a standard UI modal with duration selector, not an AI confirmation as the spec suggested.

## No test coverage

No test files were added or modified in this diff. The overlap algorithm, `getNextDueDate` (especially the ignored `days` parameter), template JSON parsing/validation, and autopilot action validation are all good candidates for unit tests.
