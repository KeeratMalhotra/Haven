# Sprint 5 bug fixes and calendar editing

This change fixes six user-facing bugs (Pomodoro auto-start, invisible timer text, broken ambient audio, chat scroll, dashboard blank state, mic error messages) and adds two features (client-side AI heuristic suggestions, calendar event editing with drag-to-reschedule). The approach is entirely frontend: no backend changes, no new dependencies beyond the already-installed `@dnd-kit/core`. Local state is updated optimistically and API calls are fire-and-forget fallbacks throughout.

**Watch for:** Drag-to-reschedule slot ID parsing is brittle and will misparse if the ID format ever gains segments (confirmed). AudioContext lifecycle has a potential leak when the ambient sound type is changed rapidly (likely). The local heuristic suggestions fire without deduplication, so a user viewing multiple tasks without deadlines will get spammed (likely).

## High-level view

The Pomodoro timer introduces a "selecting" phase that gates the countdown behind an explicit duration choice. The old external-URL ambient audio was replaced with a Web Audio API system that generates noise buffers in-memory and shapes them through filter chains. The AudioContext is created/destroyed on ambient type or pause state change, but cleanup uses a ref rather than a closure-captured variable, creating a race window on rapid toggling.

The chat scroll fix replaces the absolute-positioned composer with a flex column layout. In conversation mode, the message list is `flex-1 overflow-y-auto` and the composer sits in normal flow below it. The transition between greeting and conversation states lost its animated crossfade — `AnimatePresence` was removed from this switch.

The dashboard wraps its `Promise.all` data fetch in `.catch()` and `.finally()`, guaranteeing the page renders even on total API failure. The CommandPalette and its `Cmd+K` listener are removed, replaced by a Quick Actions FAB.

Calendar event editing uses a modal opened by clicking any event. Drag-to-reschedule integrates `@dnd-kit` with a droppable time-slot grid. The slot ID encoding (`slot-YYYY-MM-DD-HH`) is parsed by splitting on hyphens, which is fragile — one refactor away from silent data corruption. Both edit and drag paths compare events by reference equality rather than ID, so optimistic updates will silently fail after any array rebuild.

The client-side AI suggestions fire synchronously inside `reportAction` before the debounced backend call. There is no deduplication or rate-limiting beyond the 12-second auto-dismiss, so repeated actions produce a wall of identical toasts.

<summary>Issues (5)</summary>

1. **Drag slot ID parsing is fragile** (confirmed) — `handleDragEnd` splits the droppable ID on `-` and assumes the last segment is the hour. This works for `slot-2025-01-15-09` but will silently corrupt data if the format gains another segment (e.g., minutes for 30-min slots). Consider a delimiter that doesn't collide with date formatting, like `slot:2025-01-15:09`.

2. **AudioContext leak on rapid ambient switching** (likely) — The cleanup function reads from `ambientNodesRef.current`, but the ref is mutated between cleanup and setup. If React batches two rapid state updates, a transient AudioContext can survive without cleanup. Capture nodes in a local variable inside the effect and close that in the cleanup closure.

3. **Heuristic suggestions have no deduplication** (likely) — `reportAction("task_viewed", { ... })` with no deadline fires a toast every invocation. Viewing a task list with 10 deadline-less tasks produces 10 identical toasts. Add a seen-set or cooldown keyed on `${actionType}-${actionData.id}`.

4. **Event identity uses reference equality** (confirmed) — `setEvents(prev => prev.map(e => e === selectedEvent ? ...))` compares by object reference. After any refetch or concurrent state update that rebuilds the array, the match fails silently and the edit is lost. Use `e.id === selectedEvent.id`.

5. **`updateCalendarEvent` sends auth token in request body** (confirmed) — Consistent with the project's existing pattern in `api.ts`, but for a new endpoint this was an opportunity to use an `Authorization` header. Any middleware logging request bodies captures the token.

## Drag-to-reschedule slot ID fragility

The droppable time slots encode identity as `slot-${format(day, "yyyy-MM-dd")}-${hour.toString().padStart(2, "0")}`. The drag-end handler decodes with:

```ts
const parts = droppedId.split("-");
const hour = parseInt(parts[parts.length - 1], 10);
const dayStr = parts.slice(1, parts.length - 1).join("-");
```

This reconstructs `2025-01-15` because the date has exactly three hyphen-separated segments. The pattern is implicit — if the slot ID format gains another segment (minutes for finer granularity), the hour parse grabs the wrong segment and events get rescheduled to the wrong time with no error.

The same handler uses `e === draggedEvent` for the state update — reference equality that fails after any array reconstruction. `handleSaveEdit` has the identical pattern.

## AudioContext lifecycle race

The ambient audio effect depends on `[active, ambient, paused, phase]`. Cleanup calls `stopAmbientAudio(ambientNodesRef.current)` and sets the ref to null. If React batches two rapid state updates (user double-clicks between ambient options), the cleanup for the first transition interleaves with setup for the second. The ref assignment `ambientNodesRef.current = nodes` happens after `startAmbientAudio` returns, but cleanup reads from the ref at a different time — a new AudioContext can survive without cleanup.

The fix is to capture nodes locally:

```ts
useEffect(() => {
  const nodes = startAmbientAudio(ambient, volume);
  ambientNodesRef.current = nodes;
  return () => stopAmbientAudio(nodes);
}, [active, ambient, paused, phase]);
```

The current code also has an eslint-disable for exhaustive-deps masking a dependency mismatch with `volume`. If volume and ambient change simultaneously, the ordering between the volume-only effect and the audio-recreating effect is undefined.

## Client-side heuristic deduplication gap

`reportAction` fires a toast every time conditions match. The `task_viewed` check (`!actionData.due && !actionData.deadline`) has no memory of prior firings. A dashboard that reports each task produces N identical "no deadline" toasts simultaneously. The 12-second auto-dismiss prevents infinite accumulation but doesn't prevent the initial burst.

A Set tracking `${actionType}-${actionData.id}` with a 5-minute TTL would suppress duplicates without blocking legitimate new suggestions.

## Chat greeting-to-conversation transition

The old layout used `AnimatePresence mode="wait"` to crossfade between greeting and conversation states. The new layout uses a conditional render (`hasMessages ? ... : ...`) without `AnimatePresence` wrapping the branch switch. The first message causes an instantaneous layout jump rather than an animated transition. This is a UX regression from the previous behavior if the animation was intentional.
