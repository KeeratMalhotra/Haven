# Premium UI rebuild sprint: matte palette, task CRUD, Pomodoro, Spotify, empty state

Five features ship in a single sprint branch, all frontend-only. The visual system moves from glassmorphism to Notion-style matte surfaces (softer dark backgrounds, reduced shadows, no hover-lift or scale animations). Task management gains delete, priority labels, subtasks, and localStorage persistence. Focus mode becomes a Pomodoro countdown with a growing-plant SVG, ambient audio via HTML5 Audio, and session stats. A draggable Spotify mini-player floats over the dashboard. The empty-state dashboard shows quick-action cards, AI suggestion chips, and an onboarding checklist.

**Watch for:** localStorage acting as source of truth for tasks can silently diverge from the API (confirmed); the `focusUsed` checklist item reads localStorage on every render without memoization or state, creating a hydration mismatch risk on SSR (confirmed); ambient audio URLs point to third-party CDN with no fallback or license attribution in the bundle (confirmed); the Spotify embed iframe loads unconditionally with a hardcoded playlist URL that may break (confirmed).

## High-level view

The palette rework is internally consistent across CSS variables, Tailwind config, and component styles. The one concern is that `hover-lift` now sets `background-color` instead of `translateY`, making the class name semantically misleading and silently breaking any downstream code relying on the lift behavior.

Task CRUD introduces a split-brain data strategy: localStorage is the primary store, API is a fallback-on-first-load. If the backend ever writes tasks independently (another client, a webhook, an AI action), the localStorage copy wins and those changes disappear.

The Pomodoro overhaul's main risk is the audio effect's dependency array: changing `volume` triggers a full effect re-run that calls `play()` again, which may cause audible glitches in Safari.

The Spotify mini-player's snapped-state button uses `translateX(4px)` which partially clips it off-screen, leaving a narrow tap target. The cross-component coordination via synthetic `StorageEvent` is fragile and undocumented.

The empty-state dashboard has a hydration mismatch: `focusUsed` reads localStorage as an IIFE at render time, diverging from the server-rendered `false`. The identical pattern is solved correctly for `checklistDismissed` two lines above.

## Palette migration and the `hover-lift` semantic gap

The `.glass` and `.glass-strong` classes remain but are substantially toned down (blur 16px/24px → 8px/12px, saturation filter removed). Any component using `glass` will look different but won't break.

The `hover-lift` class was `translateY(-2px)` + shadow, now it's `background-color: var(--surface-hover)`. Any code relying on the physical lift (z-stacking, scroll-snap visual cues) will stop working. The mobile media query override `hover-lift:hover { transform: none }` is now redundant since there's no transform to override.

## Task data lifecycle and localStorage persistence

The split-brain problem: if the user opens a second browser tab, mutations in Tab A won't propagate to Tab B because there's no `storage` event listener on the tasks page (unlike the Spotify player which does listen). If the backend creates tasks via an AI action, the next page load still serves the localStorage copy and server-created tasks won't appear.

The `handleAiPrioritize` "Prioritized!" feedback disappears after 2 seconds via `setTimeout` without cleanup — a minor leak if the component unmounts during that window.

## Pomodoro audio effect structure

The main audio `useEffect` depends on `[active, ambient, paused, phase, volume]`. When the user drags the volume slider, the entire effect re-runs and calls `audioRef.current.play()`. In most browsers, calling `play()` on an already-playing element is a no-op, but in Safari it can cause a brief restart or stutter. A separate `useEffect` already sets `audioRef.current.volume = volume` correctly — but it's redundant with the main effect also setting volume. Remove `volume` from the main effect's dependency array and let only the dedicated effect handle volume changes.

## Spotify mini-player expand/collapse race

The `handleExpand` function calls both `saveState` (setting React state, flipping `snapped` to `false`) and `controls.start` (animating position). The state change triggers a re-render that mounts the card variant, whose `useEffect` calls `controls.set`. Then `controls.start` also fires. These two position-setting calls race — in practice `set` runs synchronously and `start` wins, but it's an unnecessary race condition that could behave differently with React concurrent features.

## Dashboard `focusUsed` hydration mismatch

`focusUsed` is an IIFE that reads `localStorage.getItem("chronai-pomodoro-stats")` during render. On the server, `typeof window === "undefined"` returns `true` so it's `false`. On the client, it may be `true`, causing a React hydration warning and visual flash. The fix: move it into `useState(false)` + `useEffect` — the same pattern used for `checklistDismissed` directly above it.

## Synthetic StorageEvent fragility

The settings page dispatches `new StorageEvent("storage", { key: "chronai-spotify-connected", newValue: ... })` to notify the mini-player. The native `storage` event normally only fires in other tabs; the manual dispatch simulates same-tab reactivity. If any future code calls `localStorage.setItem("chronai-spotify-connected", ...)` without the synthetic dispatch, the mini-player won't react. The pattern is undocumented.
