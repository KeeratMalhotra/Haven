## Audio effect closure and the suppressed lint rule

When the main effect re-runs due to `ambient` changing (user picks a new sound), it sets `audioRef.current.volume = volume`. At that moment, `volume` is whatever the closure captured on the last render that triggered the effect â€” which may be stale if the user changed the slider between the `ambient` change and the current render. In practice this is a sub-frame race that the dedicated volume effect immediately corrects on the same render cycle. The assignment is dead weight but not a bug.

## Deferred issues assessment

The three deferred issues remain unchanged in the code:

- **Split-brain (localStorage vs API):** localStorage is read first, API data is discarded if localStorage has tasks. If the backend writes tasks independently, those changes disappear on next page load.
- **Hardcoded Spotify playlist:** `37i9dQZF1DX3rxVfibe1L0` is still baked in. If the playlist goes private or gets removed, the iframe renders an error with no user recourse.
- **Audio CDN fallback:** Pixabay URLs have no fallback. `play().catch()` swallows failures silently â€” users get silence with no indication of why.

None are blocking for this sprint scope.
