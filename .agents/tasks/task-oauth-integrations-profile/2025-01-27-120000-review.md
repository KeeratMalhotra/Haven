# Incremental OAuth integrations and profile editing for ChronAI Settings

This change replaces the localStorage-based integration toggles in the Settings page with real OAuth flows backed by a new FastAPI router. Each Google service (Calendar, Tasks, Gmail, Slides) and Spotify gets its own connect/disconnect button that triggers an actual OAuth consent screen and stores granted tokens per-user in Firestore. A profile section is added with editable display name and timezone. The Spotify mini player button gets a visual refresh (smaller, matte, emerald indicator).

**Watch for:** The OAuth `state` parameter carries the user ID in plaintext without a CSRF nonce, meaning the callback endpoint trusts an attacker-controllable value to decide which Firestore document receives the tokens (confirmed). Disconnect does not revoke tokens at the provider — it only deletes them from Firestore, leaving the grant active in the user's Google/Spotify account (confirmed). Auth tokens are passed as query parameters across all integration endpoints, which means they appear in server logs and browser history (confirmed — follows existing project pattern but is especially sensitive here given the token exchange flows).

## High-level view

The backend introduces seven endpoints under `/api/integrations` that handle the full OAuth lifecycle: generating consent URLs with per-service scopes, exchanging authorization codes for tokens, persisting them in Firestore, and returning connection status. The callback endpoints accept the `state` parameter as the sole proof of user identity, bypassing all authentication. This is the most significant security gap.

The frontend Settings page is substantially rewritten. It fetches real integration status on mount, opens OAuth popups, and re-fetches status on window focus to detect when the popup completes. Profile editing piggybacks on the existing preferences endpoint. Empty `catch` blocks throughout the handlers mean all OAuth and network failures are silently swallowed with no user feedback beyond a spinner that resets.

The Spotify mini player button polish is a straightforward CSS change with a connected-indicator dot that always renders regardless of actual connection state.

<summary>Issues (7)</summary>

1. **OAuth state lacks CSRF protection** — The `state` parameter is `{user_id}:{service}` with no unguessable nonce. An attacker who knows a user's Google `sub` can craft a callback URL that binds their own authorization code to the victim's Firestore record. Add a server-generated random token stored in a short-lived cache (or signed JWT) and verify it in the callback.
2. **Callback trusts state for Firestore writes without authentication** — The `/callback` and `/spotify/callback` endpoints do not verify the caller's identity. Any request with a valid `code` and a crafted `state` writes tokens to the corresponding user document. The callback should verify that the request originated from a legitimate OAuth flow (via the nonce above).
3. **Disconnect does not revoke tokens** — `disconnect_service` and `spotify_disconnect` delete tokens from Firestore but never call the Google/Spotify revocation endpoints. The grant remains active in the user's account, and a leaked token remains usable.
4. **Silent error swallowing in frontend** — All `catch` blocks in the OAuth handlers are empty. Failures in `connectService`, `disconnectService`, `getSpotifyAuthUrl`, `disconnectSpotify`, and `handleSaveProfile` produce no user-visible feedback. At minimum, surface a toast or inline error.
5. **Spotify connected dot always renders** — The emerald indicator in `SpotifyMiniPlayer.tsx` is unconditional. It shows a "connected" dot even when the user has never connected Spotify. Gate it on the `connected` state variable.
6. **No token refresh logic** — Stored `access_token` and `refresh_token` are written once and never refreshed. When the access token expires (typically 1 hour for Google), the `connected` flag will still read `true` but API calls using those tokens will fail silently.
7. **Dead callback page** — `frontend/app/dashboard/settings/callback/page.tsx` is never reached in the current flow because the backend redirects directly to `/dashboard/settings`. Remove it or wire it into the redirect URI.

## OAuth state as an unauthenticated trust boundary

The `state` parameter in both Google and Spotify OAuth flows carries only `{user_id}:{service}` (or just `user_id` for Spotify). When the callback fires, the endpoint splits the state, extracts the user ID, and writes tokens directly to that user's Firestore document:

```python
user_id, service = parts  # from state param
# ...
user_ref = db.collection("users").document(user_id)
await user_ref.set({f"connected_services.{service}": service_data}, merge=True)
```

There is no verification that the browser presenting this callback actually belongs to `user_id`. The standard mitigation is to embed a cryptographically random nonce in the state, store it server-side (keyed to the authenticated session), and reject any callback where the nonce doesn't match. Without this, two attack vectors exist:

1. An attacker initiates their own OAuth flow, obtains a valid `code`, then replays the callback URL with the victim's `user_id` in the state — binding attacker-controlled tokens to the victim's document.
2. An attacker crafts a callback link with the victim's user ID and a code obtained through social engineering, accomplishing the same result via link injection.

The Spotify callback is identical in structure (`user_id = state`) with the same gap.

## Token lifecycle gaps

Tokens are persisted once on callback and never refreshed. The stored `expires_in` value is a snapshot from the moment of exchange — it isn't used to compute expiry time and there's no background refresh mechanism. After the access token expires (1 hour for Google, 1 hour for Spotify), subsequent API calls using these tokens will fail. The `/status` endpoint still reports `connected: true` because it checks the `connected` boolean flag rather than token validity.

Disconnect removes entries from Firestore but never calls:
- Google: `https://oauth2.googleapis.com/revoke?token={token}`
- Spotify: `https://accounts.spotify.com/api/token` with revoke semantics

This means disconnecting is purely cosmetic from the provider's perspective. If tokens were ever exfiltrated, they remain usable until natural expiry or manual revocation by the user in their account settings.

## Frontend error handling

Every OAuth handler in the Settings page catches errors with an empty block:

```typescript
const handleConnectService = async (service: string) => {
  // ...
  try {
    const { auth_url } = await connectService(authToken, service);
    const popup = window.open(auth_url, "_blank", "width=600,height=700,popup=yes");
    oauthPopupRef.current = popup;
  } catch {
    setConnectingService(null);
  }
};
```

If the backend is unreachable, returns a 401 (token expired), or the popup is blocked by the browser, the user sees only a spinner that resets. The same pattern appears in `handleDisconnectService`, `handleConnectSpotify`, `handleDisconnectSpotify`, and `handleSaveProfile`. Given that OAuth popups are frequently blocked by default in modern browsers, this is a likely failure mode with no diagnostic path for the user.

## Spotify mini player indicator unconditionally renders

In `SpotifyMiniPlayer.tsx`, the emerald dot is rendered without any condition:

```tsx
<span className="absolute -top-0.5 -right-0.5 h-2 w-2 rounded-full bg-emerald-400 ring-2 ring-gray-900/80" />
```

The `connected` state variable exists in the component (initialized from localStorage) but this dot is outside any conditional. A new user who has never connected Spotify sees an active-looking green indicator on the collapsed player button.

## Dead callback page and race on focus refetch

The backend's Google callback redirects to `{FRONTEND_ORIGIN}/dashboard/settings?connected={service}`, bypassing the callback page at `/dashboard/settings/callback/page.tsx` entirely. That page would only activate if `GOOGLE_REDIRECT_URI` pointed at the frontend rather than the backend, which isn't the case. It's dead code as currently wired.

The focus-based refetch strategy waits 1 second after the window regains focus to re-query integration status. If the OAuth callback hasn't completed server-side processing within that window (slow token exchange, Firestore write latency), the status poll returns stale data. The popup reference is then nulled out, so no subsequent refetch attempt occurs. This is a possible race under slow network conditions — a retry with exponential backoff (or polling until status changes) would be more robust.
