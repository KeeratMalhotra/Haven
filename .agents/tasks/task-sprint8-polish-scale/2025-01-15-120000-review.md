# Analytics dashboard, mobile responsiveness, and polish pass

This sprint adds a new `/dashboard/analytics` page with SVG-only charting (bar, line, ring, and score widgets), converts all modals and the AI chat panel to full-screen on mobile, introduces Framer Motion page transitions between dashboard routes, applies touch-target minimums and responsive typography, fixes hydration mismatches (time indicators, random greetings, confetti), adds a `useDebounce` hook and `memo`-wrapped Kanban components for performance, and adds Open Graph / Twitter Card meta plus favicon verification. It also lazy-loads the heavy FocusMode and AutoPilotPanel components via `next/dynamic`.

**Watch for:** The `useDebounce` hook is created but never imported anywhere (confirmed dead code). The SVG `linearGradient` in the line chart uses a hardcoded `id="lineGradient"` which will collide if the component is ever rendered more than once on the same page (likely issue given current single-instance usage, but fragile). The full-screen modal on mobile removes `max-w-lg` and sets `h-full` with `overflow-y-auto`, but the modal's content animation (`scale: 0.96, y: 8`) was designed for a centered card — on mobile it animates a full-viewport sheet from slightly below center, which is visually jarring compared to a bottom-sheet slide-up (possible concern, depends on design intent).

## High-level view

The analytics page reads from three localStorage keys, computes daily aggregates over a selectable time window, and renders four SVG chart widgets. The empty-state gate checks for the presence of the data arrays, not for any data within the selected time window — so the page shows all-zero charts when data exists but falls outside the selected period.

The mobile responsiveness strategy combines Tailwind responsive prefixes (`md:`) with a JavaScript `matchMedia` listener in the calendar page. The matchMedia approach is needed because the calendar changes data-fetching behavior (fewer days), not just layout — but it creates a coupling between the CSS breakpoint (`md: 768px`) and the JS breakpoint (`max-width: 767px`) that could drift if the Tailwind config is customized.

Page transitions use `AnimatePresence` in the dashboard layout keyed on `pathname`, respecting `prefers-reduced-motion` via Framer's `useReducedMotion` hook. However, individual pages (analytics, dashboard home, etc.) still run their own stagger animations unconditionally — reduced-motion is only honored for the inter-page transition, not the intra-page entrance.

The hydration fixes move `Date.now()` and `Math.random()` calls into `useEffect` with deterministic server-side fallbacks. The confetti particles use modular arithmetic instead of randomness.

Performance work includes `memo()` on three Kanban sub-components, `useMemo` on filtered task lists, and `dynamic()` imports for FocusMode and AutoPilotPanel. The `useDebounce` hook was written but never wired up to any input — the debounce intent from the sprint requirements is unfulfilled.

## Analytics SVG gradient collision

The `LineChart` SVG defines a `<linearGradient id="lineGradient">` in its `<defs>` section. Because SVG IDs are document-global, rendering two `LineChart` components on the same page would cause the second to reference the first's gradient definition. Currently only one instance exists, but the component is written as a reusable function — the ID should use React's `useId()` or accept a prop-based suffix to prevent silent visual breakage if reuse occurs.

```tsx
<linearGradient id="lineGradient" x1="0" y1="0" x2="0" y2="1">
```

## Modal full-screen behavior on mobile

The Modal uses `items-end` on mobile (aligning content to the bottom) combined with `h-full w-full rounded-none` — a full-viewport sheet. The animation variants still use `scale: 0.96, y: 8`, a center-card entrance. On a full-screen sheet anchored to the bottom, this reads as a zoom from the middle rather than a sheet reveal. A conditional variant that uses `y: "100%"` on mobile and the existing `scale + y: 8` on desktop would match user expectations for each form factor.

## Calendar matchMedia breakpoint

The calendar page uses `window.matchMedia("(max-width: 767px)")` to determine mobile state, which drives data fetching (day count) and view override (week → day). The magic number 767 corresponds to `md: 768px` in Tailwind's default config. If the project ever customizes the `md` breakpoint in `tailwind.config.ts`, the CSS layout and the JS behavior will disagree — the layout could be "mobile" while the JS still fetches desktop data, or vice versa.

## Intra-page animations and reduced motion

The layout correctly gates page transitions on `useReducedMotion()`. But the analytics page defines its own:

```tsx
const containerVariants = {
  hidden: { opacity: 0 },
  visible: { opacity: 1, transition: { staggerChildren: 0.08, delayChildren: 0.1 } },
};
```

These run unconditionally. A user who sets `prefers-reduced-motion: reduce` will see no page-to-page slide, but will still see staggered fade-in/translate-up animations inside each page. The fix is to either pass `useReducedMotion()` down from the layout or check it within each page's motion config.

## Touch target desktop reset

Touch targets use `min-h-[44px] min-w-[44px]` on mobile with `md:min-h-0 md:min-w-0` to reset on desktop. The reset to `0` means elements like the 14px trash icon button rely solely on padding for their hit area on desktop. This isn't a mobile concern (those get 44px), but the desktop hit area for the task delete button becomes whatever the icon size plus minimal padding provides — potentially uncomfortably small for trackpad users.
