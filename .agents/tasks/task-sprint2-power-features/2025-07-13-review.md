# Sprint 2 Power Features: Priority Engine, Smart Scheduling, Subtask Decomposition, Command Palette, Focus Mode

This change adds five features at once: a new PriorityAgent that ranks tasks via Gemini, smart time-slot suggestions and focus-session calendar blocking on the SchedulerAgent, a decompose action on the PlannerAgent that splits goals into max-6 subtasks, a frontend Cmd+K command palette built on cmdk, and a full-screen FocusMode timer overlay with countdown ring. The orchestrator's routing table expands to direct priority/focus/decompose intents to the right agent, and a new `GET /api/priorities` REST endpoint surfaces the ranking externally. Nine new test classes cover the backend additions.

**Watch for:** The priorities endpoint passes the auth token as a query parameter, exposing it in server logs and browser history (confirmed). The `onSendReady` callback pattern creates a timing window where the command palette's send function may reference a stale or unregistered closure (likely). Focus session creates a calendar event starting "now" but the FocusMode frontend component is entirely client-local with no backend round-trip, so the two clocks can drift (confirmed).

## High-level view

The PriorityAgent follows the same pattern as the other agents: Gemini prompt, MCP data fetch, JSON response with fallback. The structured fallback when Gemini fails simply echoes tasks in arrival order with "medium" urgency, which is honest but provides no value over showing the raw list.

Smart scheduling slots a new `suggest_time` action into the existing SchedulerAgent dispatch. It fetches free slots, scores them by time-of-day preference, then stores a pending confirmation so the user can approve the block. The scoring is simple (fixed point values for hour ranges) and doesn't actually check for post-meeting buffers despite the prompt claiming it.

Subtask decomposition reuses the PlannerAgent's MCP-create path, truncating Gemini's response to 6 items before creating tasks. The tasks are always created immediately, with no dry-run or confirmation step, which means a bad Gemini response materialises as real Google Tasks the user has to clean up.

The command palette is a clean cmdk integration with grouped actions, keyboard shortcut, and Framer Motion overlay. It communicates with chat via a ref-based `onSendReady` callback that the parent dashboard threads through, which works but introduces a subtle ordering dependency on React effect execution.

FocusMode is a self-contained client-side timer with no backend persistence. When the backend's `focus_session` action blocks the calendar and the frontend shows the timer, there's no shared state — pausing the frontend timer doesn't unblock the calendar event, and closing the browser silently abandons the session.

<summary>Issues (7)</summary>

1. **Auth token in query string** — `GET /api/priorities?auth_token=...` leaks the token into URL logs, referer headers, and browser history. Move to an `Authorization` header or POST body.
2. **Post-meeting buffer not implemented** — The `_pick_optimal_slot` scorer claims to avoid post-meeting slots but only checks hour-of-day, never consulting the event list. Either implement the buffer check or remove the claim from the prompt.
3. **No decompose confirmation step** — Subtasks are created in Google Tasks immediately without user confirmation. A bad Gemini response creates garbage tasks the user must manually delete.
4. **`onSendReady` timing race** — The `useEffect` that exposes the send function fires after render; if `CommandPalette` triggers an action before the effect runs (e.g., programmatic open on mount), the ref is null.
5. **Frontend/backend focus session disconnect** — FocusMode timer runs client-side only. Pausing or stopping the timer doesn't modify the calendar block, and there's no heartbeat or completion callback to the backend.
6. **Priority endpoint missing `user_id` propagation on auth failure** — If `verify_google_token` returns a dict without `"sub"` (partial token info), `user_id` becomes empty string and `get_user_context("")` returns generic context, silently degrading personalisation.
7. **Decompose creates tasks even when MCP partially fails** — Individual `create_task` failures are swallowed with `pass`; the response count may show fewer created tasks but gives no list of which failed, making it hard for the user to reconcile.

## Auth token exposure on the priorities endpoint

The new `GET /api/priorities` endpoint accepts `auth_token` as a query parameter:

```python
@app.get("/api/priorities")
async def get_priorities(auth_token: str = ""):
```

Every other sensitive endpoint in this app uses the same pattern (query param auth), so this is consistent — but it's still a security concern for a REST endpoint that browsers or front-end code will hit directly. The token appears in access logs, proxy logs, browser history, and `Referer` headers on any subsequent navigation. For an endpoint likely called via `fetch`, the token should travel in an `Authorization: Bearer ...` header or as a POST body field.

**Confidence: confirmed** — the endpoint definition is in the diff at `main.py:712`.

## Post-meeting buffer: claimed but not enforced

The `SCHEDULER_PROMPT` states "Avoid slots immediately after meetings (15 min buffer)" and the `SUGGEST_TIME RULES` section repeats it. However, `_pick_optimal_slot` only scores by hour-of-day:

```python
if preferred_time == "morning":
    if 9 <= hour <= 11:
        score += 10
```

There is no code path that fetches the user's events and penalises slots adjacent to meetings. The prompt instructs Gemini to consider buffers, but the actual slot selection is deterministic Python code that ignores meeting proximity entirely.

**Confidence: confirmed** — `_pick_optimal_slot` receives only the free-slot list, not the events list, and never calls `_list_events`.

## Subtask decomposition creates without confirmation

When `action == "decompose"`, the planner immediately loops through `subtasks` and calls `create_task` for each:

```python
for task_item in subtasks:
    result = await self.call_mcp_tool("google-tasks", "create_task", {...})
    created_tasks.append(result)
```

There's no intermediate "Here are the subtasks I'd create — should I proceed?" confirmation, unlike the scheduler's `suggest_time` which stores a pending action for user approval. If Gemini hallucinates irrelevant subtasks, they become real tasks in the user's Google Tasks immediately.

This also means partial MCP failures (individual tasks silently swallowed with `except: pass`) produce an inconsistent state where some subtasks exist and others don't, with no clear indication to the user which ones failed.

**Confidence: confirmed** — lines 413–424 of `planner.py`.

## `onSendReady` callback and stale-ref risk

The dashboard page stores the chat send function in a ref:

```tsx
const sendToChatRef = useRef<((msg: string) => void) | null>(null);
const handleSendReady = useCallback((sendFn: (content: string) => void) => {
  sendToChatRef.current = sendFn;
}, []);
```

Inside `ChatExperience`, a `useEffect` publishes the function:

```tsx
useEffect(() => {
  onSendReady?.((content: string) => send(content, "chat"));
}, [send, onSendReady]);
```

The `send` function comes from the `useChatSocket` hook, which recreates it when the WebSocket reconnects. Each reconnection triggers a new `onSendReady` call, updating the ref. But during the brief window between reconnect and effect execution, the ref holds a closure over the old (disconnected) socket. For normal user interaction this is invisible, but programmatic triggers (e.g., URL-driven command palette open) could fire into the void.

**Confidence: likely** — the window exists structurally; whether a user hits it in practice depends on timing.

## Frontend FocusMode is entirely disconnected from the backend focus session

The backend `_do_focus_session` creates a calendar event starting at `now_ist().isoformat()`. The frontend `FocusMode` component runs a `setInterval` countdown independently. There is no shared state:

- Pausing the frontend timer doesn't shorten or remove the calendar block.
- Stopping early doesn't update the calendar event duration.
- If the user refreshes the page, the timer is lost but the calendar block remains.
- Completing the timer doesn't send a "completed" signal anywhere.

This means the calendar always shows the full original block regardless of what the user actually does. For a focus timer that's meant to integrate with calendar blocking, the two halves operate in isolation.

**Confidence: confirmed** — `FocusMode.tsx` has no network calls; `_do_focus_session` has no WebSocket push to the frontend.

## Test coverage

The nine new test classes cover the happy path for each feature: priority ranking order, empty state, Gemini fallback, suggest-time morning preference, no-slots-available, focus session calendar creation, default duration, subtask decomposition, and the 6-subtask cap.

Not tested: the `GET /api/priorities` endpoint itself (no integration test hitting the route), the `_pick_optimal_slot` edge cases (all slots scoring zero, empty slot list with error dicts), the `onSendReady` lifecycle, or any interaction between the command palette triggering focus mode and the backend focus session path.
