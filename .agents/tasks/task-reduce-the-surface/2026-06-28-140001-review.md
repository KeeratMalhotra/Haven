# Reduce the Surface: grouped sidebar nav + universal quick capture (v1)

Sprint 13 reshapes the dashboard's navigation and adds a frictionless capture path without touching any backend. The sidebar is split into a calm primary group (Dashboard, Tasks, Calendar), a de-emphasized collapsible "More" group (Planner, Habits, Analytics), and Settings pinned at the bottom, with a single `renderNavItem` helper so active/hover/transition states are uniform everywhere. A new `QuickCapture` command modal opens from the TopBar "+" or the `n` key, parses natural language through the existing braindump backend, and confirms with a toast. The braindump endpoint and FastAPI side are reused unchanged. Build, lint, and the 3 test suites (9 tests) all pass clean.

**Watch for:**
- The "More" toggle is a visual no-op while you're on a secondary route â€” `moreExpanded = moreOpen || onSecondaryRoute` keeps the group open regardless of the button, so clicking "Collapse more navigation" appears to do nothing (confirmed).
- `n` opens QuickCapture even when another overlay (chat panel, a non-input modal) is already open; the guard only blocks editable-element focus, not existing modals (likely).
- Empty-result and thrown-error paths from capture are handled gracefully (modal stays open, warning toast), but neither path is covered by tests (confirmed).

## High-level view

The information architecture change is purely a regrouping: the old flat `NAV_ITEMS` array is split into `PRIMARY_ITEMS`, `SECONDARY_ITEMS`, and a standalone `SETTINGS_ITEM`, all rendered through one shared helper. Every one of the seven destinations stays reachable in both expanded and collapsed modes, so nothing was removed. The "More" open/closed state persists to localStorage and auto-expands when the active route lives in the secondary group â€” that auto-expand has a small interaction quirk with the manual toggle.

Quick capture is a self-contained modal mounted inside `AIContextProvider`. Because the TopBar lives outside that provider, the "+" button reaches the modal through a `chronai-open-quick-capture` window event, mirroring the established `chronai-open-chat` / `chronai-start-focus` pattern. Submission reuses `parseBraindump` and surfaces the result as a toast through the existing AI notification channel.

The `n` shortcut guard bails on any modifier key and on focus inside input/textarea/select/contenteditable, checking both the event target and `document.activeElement`. The remaining gap is that it doesn't consider whether another modal is already open.

Failure handling is fail-soft: empty results and network/parse errors both keep the modal open with a warning toast so the user can retry without retyping. The test suite covers the open paths and the happy submit, but not the empty or error branches, nor the sidebar regrouping behavior.

The diff also carries FEAT-001, a test-infra baseline (ESM mocks for `react-markdown`/`remark-gfm`) unrelated to the UX work but a justified prerequisite for the new component test to run.

### Sidebar IA: primary group, collapsible More, pinned Settings

The single `NAV_ITEMS` array becomes `PRIMARY_ITEMS` (Dashboard, Tasks, Calendar), `SECONDARY_ITEMS` (Planner, Habits, Analytics), and a standalone `SETTINGS_ITEM`, and the duplicated per-link JSX is lifted into one `renderNavItem` helper that owns the active indicator, hover states, collapsed tooltip, and framer-motion `layoutId` indicators. That helper is what makes the "consistent navigation" requirement real â€” Settings now animates identically to the primary links instead of being a special case.

Collapsed (icon-only) mode renders `[...PRIMARY_ITEMS, ...SECONDARY_ITEMS]` directly with no disclosure, and Settings is rendered separately in the bottom section in both modes, so all seven destinations stay reachable whether expanded or collapsed.

The behavioral snag is the interaction between persisted state and auto-expand:

```ts
const onSecondaryRoute = SECONDARY_ITEMS.some((item) => isActive(item.path));
const moreExpanded = moreOpen || onSecondaryRoute;
```

While the active route is in the secondary group, `moreExpanded` is always true. The toggle still flips `moreOpen` and `aria-expanded` still reflects `moreExpanded`, so a user on `/dashboard/planner` who clicks "Collapse more navigation" sees the label change intent but the group never collapses. Letting an explicit collapse override auto-expand (or suppressing the toggle on secondary routes) would remove the dead interaction.

The secondary group is ordered Planner, Habits, Analytics; the brief listed Habits, Analytics, Planner. Cosmetic, worth a one-line confirmation the order was deliberate.

### Quick Capture flow and the CustomEvent bridge

QuickCapture is mounted inside `AIContextProvider` in the dashboard layout, while the TopBar renders outside that provider. The "+" button therefore dispatches a `chronai-open-quick-capture` window event that the modal listens for â€” the same indirection the codebase already uses for `chronai-open-chat` and `chronai-start-focus`, so it fits the existing topology rather than inventing a new one.

Reusing the braindump backend means natural-language date parsing ("call dentist tomorrow 3pm") and task/event/habit creation come for free. One consequence of the shared `(session as ...).accessToken || ""` token pattern: `parseBraindump` throws immediately on an empty token, so if a submit fires before the session has hydrated, the user gets the generic error toast rather than a capture. Unlikely on an authenticated dashboard, but it routes a transient auth-not-ready state into a "something went wrong" message.

### The `n` shortcut input guard

The guard returns early on any modifier (`metaKey`/`ctrlKey`/`altKey`) so it never competes with Cmd+K, returns early if the modal is already open, and checks both the event target and `document.activeElement` against `INPUT`/`TEXTAREA`/`SELECT`/`isContentEditable` before opening. Checking `activeElement` in addition to the target matters because synthetic or retargeted key events don't always carry the focused element as `target`. No other global single-key `keydown` listener exists in the app, so there's no shortcut collision.

The gap is that the guard reasons only about editable focus, not about other open overlays. With the AI chat panel open (and focus not in its composer) or a non-input modal showing, pressing `n` will still stack QuickCapture on top. Relatedly, QuickCapture's Escape is handled on its own input and prevents default, but the keystroke still bubbles to the document, where the chat panel's own Escape listener may also fire â€” worth confirming the two overlays don't both close on a single Escape.

### Empty and error results from capture

Both unhappy paths fail soft, which is the right call for a capture box. When the backend parses but creates nothing (`counts.tasks + counts.events + counts.habits === 0`), it shows "Couldn't capture that - try adding a time or more detail." and keeps the modal open by resetting `submitting` rather than closing. A thrown error (non-2xx, network failure, empty token) is caught, shows "Something went wrong capturing that - please try again.", and likewise leaves the modal open so the typed text survives for a retry.

### Functionality preserved

All seven nav destinations remain reachable: the three primary links, the three secondary links (via the More disclosure when expanded, or inline when collapsed), and Settings pinned at the bottom in both modes. The four TopBar quick actions (Add Task, Add Event, Start Pomodoro, Start Focus) are untouched â€” the Quick Capture entry was added above them with a divider, not in place of any. Theme toggle and collapse controls are unchanged.

### Test coverage

`QuickCapture.test.tsx` covers the two open paths (the `n` key on `document.body` and the `chronai-open-quick-capture` event), the happy submit (asserts `parseBraindump` is called with the typed text, the `Added '...'` toast fires, and the modal closes), the editable-focus guard, and the modifier-key guard.

The sidebar regrouping is the higher-risk untested area since it's where "don't remove functionality" could regress silently.
