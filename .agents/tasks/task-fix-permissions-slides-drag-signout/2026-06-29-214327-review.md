# Graceful degradation for denied permissions, false-positive connect toasts, drag duplication, and sign-out

Five fixes rolled into one commit: the NextAuth callback no longer throws when `access_token` is null (user denied scopes at consent); the dashboard breaks out of its infinite-loading state and shows a degraded banner; the Slides connect flow stops emitting false "connected successfully" toasts by snapshotting pre-connect state; drag-and-drop uses a ref guard to prevent duplicate status transitions; and a Sign Out card now exists at the bottom of Settings.

**Watch for:** The drag-processing guard resets on a 100 ms timer, which means a legitimate second drag that completes within 100 ms of the first will be silently dropped (likely). The polling loop for connect never terminates early when `preConnectStatusRef` indicates stale data—it keeps polling until `maxAttempts`, burning network cycles for 30 s even though it already knows the service was previously connected (confirmed). The dashboard's data-fetch `useEffect` gates on `!accessToken`, so in degraded mode the user sees an empty dashboard with no indication that data fetches were skipped beyond the banner (confirmed).

## High-level view

The auth resilience change coalesces `null` to `""` at the JWT and session callbacks, shifting the "no permissions" semantic downstream: every consumer of `accessToken` now has to interpret `""` as "authenticated but without scopes," and not all of them do so explicitly (the banner in the dashboard is the only explicit handling).

The slides false-positive fix snapshots connection state into a ref before opening the OAuth popup, then compares after polling. This solves the stale-data toast, but the else-branch (service already connected) continues to poll without ever clearing the interval, waiting to hit the `maxAttempts` ceiling.

The drag guard is a debounce-style ref lock with a 100 ms timer reset. It prevents re-entrant `handleDragEnd` calls from @dnd-kit sensor edge cases, but the timer-based approach silently swallows any legitimate drag that completes within that window. A synchronous reset at the end of the handler would avoid this.

## Pre-connect snapshot and polling lifecycle

The gap is in the else-if branch that handles already-connected services:

```typescript
} else if (
  connectingService &&
  status[connectingService]?.connected &&
  preConnectStatusRef.current[connectingService]
) {
  // Service was already connected before - do NOT show toast
  // Keep polling in case it disconnects and reconnects
}
```

This branch does nothing—no `clearInterval`, no state cleanup. It polls every 3 seconds until `maxAttempts` (30 s). The comment says "keep polling in case it disconnects and reconnects," but that scenario is unlikely within a 30-second window, and the user already sees the service as connected. The interval should be cleared here to avoid unnecessary network load, and `connectingService` should be reset so the button leaves its loading state immediately rather than spinning for 30 seconds.

## Drag guard timing

```typescript
dragProcessingRef.current = true;
setTimeout(() => { dragProcessingRef.current = false; }, 100);
```

The mechanism is timer-based rather than event-based. @dnd-kit guarantees one `onDragEnd` per drag sequence, but if the library or sensor misbehaves (the original bug), a second event might arrive within the same microtask queue. A synchronous reset at the end of the handler would be both simpler and less prone to edge-case swallowing:

```typescript
try {
  // ... mutation logic
} finally {
  dragProcessingRef.current = false;
}
```

This still guards against re-entrancy within the same synchronous execution (React event handlers batch), but doesn't create a time window where legitimate subsequent drags are dropped.

## Dashboard degraded experience

The data-fetch `useEffect` gates on `!accessToken`:

```typescript
if (!onboardingChecked || !accessToken) return;
```

In degraded mode, `tasks`, `events`, `habits` remain at `[]`. The dashboard widgets render as empty cards. The banner at the top explains the situation, but individual widget sections give no context about why they're empty. A per-widget "Connect your Google account to see data here" message would reduce confusion.
